# 데이트 기록 앱 — 설계서

> 여자친구와 데이트를 계획하고(갈 데이트) 추억을 남기는(간 데이트) 둘만의 웹앱.
> Vercel 배포, 공유 비밀번호로 접근, 예쁘고 가볍게.

---

## 1. 목표 & 원칙

- **둘만 본다** — 화려한 관리자 기능 X, 감성/추억 중심.
- **두 가지 상태** — `갈 데이트`(예정) / `간 데이트`(완료)를 명확히 구분.
- **자연스러운 전환** — 갈 데이트를 다녀오면 버튼 하나로 간 데이트가 됨.
- **똑똑한 일정 도우미** — 장소 사이 이동시간을 **차/대중교통 둘 다** 보여주고, 조정 판단은 너희에게.
- **간단한 운영** — 서버 관리 없이 Supabase + Vercel로 끝.

## 2. 핵심 개념

| 용어 | 뜻 |
|------|-----|
| **코스(Course)** | 하루치 데이트 = 여러 장소(Stop)의 묶음. 예) "금요일 데이트" |
| **장소(Stop)** | 코스 안의 개별 일정. 예) "신당 오신돼지갈비 16:00 예약" |
| **멤버(Member)** | 이 앱을 쓰는 사람(둘). 공유 비번 통과 후 **이름**을 정해 식별. 사진·리뷰의 작성자 |
| **상태(status)** | `planned`(갈 데이트) / `done`(간 데이트) |
| **구간(Segment)** | 연속한 두 장소 사이. **차/대중교통 이동시간을 둘 다** 계산해 보여주는 단위 (저장 X, 계산값) |

> **갈 데이트 = planned 코스**, **간 데이트 = done 코스**. 같은 테이블, `status`로만 구분.
> 그래서 "갈 → 간" 전환은 `status` 변경 + 사진 첨부 허용으로 자연스럽게 됨.

## 3. 데이터 모델 (Supabase / Postgres)

```
members (멤버 = 우리 둘)
├─ id         uuid pk
├─ name       text          -- 로그인 시 정한 이름 (변경 가능)
├─ color      text          -- 이름표 색(구분용, nullable)
└─ created_at timestamptz

courses (코스 = 하루 데이트)
├─ id           uuid pk
├─ title        text          -- "금요일 데이트"
├─ date         date          -- 2026-07-17
├─ status       text          -- 'planned' | 'done'
├─ memo         text          -- 공용 메모(둘 다 편집)
├─ cover_photo  text          -- 대표 사진 URL (nullable)
├─ updated_by   uuid fk → members.id  -- 마지막 수정자(누가 조정했는지)
├─ created_at   timestamptz
└─ updated_at   timestamptz

stops (장소 = 일정)
├─ id            uuid pk
├─ course_id     uuid fk → courses.id (on delete cascade)
├─ sort_order    int           -- 코스 내 순서 (0,1,2…)
├─ name          text          -- "신당 오신돼지갈비"
├─ place_query   text          -- 카카오 검색용 원문 (지오코딩 입력)
├─ lat           double        -- 위도 (지오코딩 결과 캐시)
├─ lng           double        -- 경도
├─ arrive_at     time          -- 도착/예약 시각 16:00
├─ stay_min      int           -- 머무는 시간(분), 선택 입력 (nullable)
├─ is_reserved   bool          -- 예약 여부
└─ memo          text          -- 장소 공용 메모

reviews (리뷰 — 서로 각자 남김)
├─ id         uuid pk
├─ course_id  uuid fk → courses.id (on delete cascade)
├─ author_id  uuid fk → members.id
├─ rating     int           -- 별점 1~5
├─ one_line   text          -- 한줄평
├─ created_at timestamptz
└─ updated_at timestamptz
   ※ UNIQUE(course_id, author_id) — 한 데이트에 사람당 리뷰 1개(수정 가능)

photos (사진 — 서로 업로드)
├─ id         uuid pk
├─ course_id  uuid fk → courses.id (on delete cascade)
├─ stop_id    uuid fk → stops.id  (nullable — 특정 장소에 귀속 시)
├─ author_id  uuid fk → members.id  -- 누가 올렸는지
├─ url        text          -- Supabase Storage public URL
├─ caption    text
├─ taken_at   timestamptz
└─ created_at timestamptz
```

**Storage 버킷**: `date-photos` (공개 읽기, 업로드는 앱 통해서만).

> 이동수단을 하나로 고정하지 않는다. 구간마다 **차/대중교통을 둘 다 계산해 보여주고**,
> 실제 조정(미루기·당기기·택시)은 너희가 판단. 그래서 transport 컬럼은 두지 않음.

## 4. 주요 기능

### 4.1 갈 데이트 (planned)
- **둘 다** 코스/장소 **추가·수정·삭제**, 순서 **드래그로 변경** 가능(공유 편집).
- 마지막으로 조정한 사람(`updated_by`) 표시 — "○○가 시간 변경함" 정도로.
- 장소 입력 시 카카오 키워드 검색 → 좌표 자동 저장.
- 구간마다 **차 / 대중교통 이동시간을 둘 다** 표시 (대중교통은 대기시간 포함).
- **여유/촉박 표시**: `다음 도착시각 − 이전 도착시각`(=간격)과 이동시간을 비교해,
  이동시간이 간격보다 크면 "촉박/지각 위험" 경고 → 미룰지·택시탈지 판단은 너희 몫.
- "다녀왔어요" 버튼 → `done`으로 전환(간 데이트로 이동).

### 4.2 간 데이트 (done)
- 갈 데이트에서 전환되거나, **처음부터 직접 생성** 가능.
- **둘 다 사진 업로드** — 각 사진에 올린 사람(`author_id`) 표시, 캡션.
- 대표 사진 지정, 공용 메모 작성.
- **서로 리뷰**: 같은 데이트에 **각자 별점+한줄평** 1개씩(수정 가능). 화면엔 "내 리뷰 / 상대 리뷰" 나란히.
- 지나간 코스도 이동시간 정보는 그대로 열람 가능(추억 기록).

### 4.3 공통 / 협업
- 공유 데이터 — 둘 중 누가 바꿔도 상대 화면에 반영(새로고침/재조회 시).
- 작성자 구분 — 사진·리뷰에 이름표(멤버 색)로 누가 남긴 건지 표시.
- 타임라인 뷰: 장소 → (이동 30분, 비는시간 3시간) → 다음 장소.
- 홈: 다가오는 갈 데이트 + 최근 간 데이트 피드.
- **설정에서 내 이름 변경** 가능 — 바꾸면 기존 사진·리뷰 작성자 표시도 함께 갱신(멤버 참조).

## 5. 이동시간 표시 로직 (핵심 UX)

> **원칙**: 앱이 이동수단을 정하지 않는다. 두 옵션을 나란히 보여주고 사람이 결정.

### 5.1 각 구간(장소[i] → 장소[i+1])에 붙는 정보
```
간격        = 장소[i+1].arrive_at − 장소[i].arrive_at          // 두 예약 사이 총 시간
차량 이동    = 카카오모빌리티(좌표 i, 좌표 i+1)                 // "차량 약 32분"
대중교통 이동 = ODsay(좌표 i, 좌표 i+1)                          // "대중교통 약 51분 (+대기 약 8분)"

각 수단별 여유 = 간격 − 이동시간(− stay_min 있으면 차감)
  여유 ≥ 0  → "여유 N분" (초록)
  여유 < 0  → "N분 촉박 / 지각 위험" (빨강)
```

### 5.2 화면에 이렇게 보인다
```
┌ 16:00  신당 오신돼지갈비  (예약)
│
│   ⏱ 다음까지 4시간
│   🚗 차량 약 30분  → 여유 3시간 30분
│   🚇 대중교통 약 50분 (+대기 8분)  → 여유 3시간
│
└ 20:00  하남 미미사 양꼬치  (예약)
```
- `stay_min`(머무는 시간)은 선택 입력. 넣으면 여유에서 자동 차감, 없으면 도착시각만으로 계산.
- 결과는 저장 안 하고 조회 시 계산(장소 바뀌면 항상 최신).
- 대중교통은 배차/대기 특성상 **대기시간을 따로 명시**해 촉박함을 더 정확히 전달.

### 5.2 외부 API 매핑
| 용도 | API | 비고 |
|------|-----|------|
| 장소명 → 좌표 | **카카오 로컬**(키워드 검색) | 무료, `stops.lat/lng`에 캐시 |
| 차 이동시간 | **카카오모빌리티 길찾기** | 무료 티어, 자동차 소요시간 |
| 대중교통 이동시간 | **ODsay 대중교통 길찾기** | 무료 티어, 환승·소요시간 |

> API 키는 **서버(Next.js Route Handler / Server Action)** 에서만 사용 → 클라이언트 노출 금지.
> 좌표는 한 번 구하면 캐시하므로 지오코딩 호출은 장소 추가/수정 시에만.

### 5.3 제공된 스케줄 기준 예시 (개략 — 실제값은 API가 확정)
**금요일** (16:00 신당 → 20:00 하남, 간격 4시간)
- 차량/대중교통 어느 쪽이든 여유 큼 → 초록

**토요일**
- 11:00 강동 → 13:00 명동 (간격 2시간): 강동→명동 이동시간 고려 시 다소 촉박 가능 → **경고 후보**
- 13:00 명동 → 18:30 용산 (간격 5시간 30분): 여유 큼 → 초록

## 6. 화면 구성

| 경로 | 화면 | 내용 |
|------|------|------|
| `/login` | 비밀번호 + 이름 | 공유 비밀번호 입력 → **내 이름 정하기**(멤버 생성/선택) → 쿠키 세팅 |
| `/` | 홈 | 다가오는 갈 데이트 + 최근 간 데이트 피드 |
| `/plans` | 갈 데이트 목록 | planned 코스 카드 |
| `/plans/[id]` | 갈 데이트 상세 | 타임라인 + 이동/비는시간 + 편집 |
| `/memories` | 간 데이트 목록 | done 코스 갤러리 |
| `/memories/[id]` | 간 데이트 상세 | 타임라인 + 사진 갤러리 |
| `/new` | 코스 생성 | planned/done 선택 후 장소 추가 |
| `/settings` | 설정 | **내 이름 변경**, 멤버 색, 로그아웃 |

- 모바일 우선(둘 다 폰으로 볼 것). 감성 톤: 부드러운 색, 큰 사진, 카드형 타임라인.

## 7. 기술 스택

- **프론트/백**: Next.js (App Router) — Vercel 배포
- **DB & 스토리지 & 인증세션**: Supabase (Postgres + Storage)
- **지도/이동**: 카카오 로컬·모빌리티 + ODsay(대중교통)
- **스타일**: Tailwind CSS (+ 필요 시 shadcn/ui)
- **상태**: 서버 컴포넌트 + 최소 클라이언트 상태

---

## 8. 접근 / 보안

- **공유 비밀번호 1개 + 이름**: `/login`에서 비번 입력 → 서버에서 환경변수 해시와 비교 → 통과 시
  **이름 입력**(기존 멤버 선택 또는 새 멤버 생성) → HttpOnly 쿠키에 `authed` + `member_id` 저장.
- 미들웨어에서 쿠키 없으면 `/login`으로 리다이렉트.
- 인증은 "우리 둘"을 막는 게 아니라 외부인 차단용 — 비번은 공유, 이름으로만 서로 구분.
- 이름 변경은 `members.name` 업데이트 → 참조하는 사진·리뷰 표시가 자동으로 따라감.
- Supabase는 **service role 키를 서버에서만** 사용(또는 RLS + anon 정책). 클라이언트엔 anon key만.
- API 키(카카오/ODsay/Supabase service)는 전부 Vercel 환경변수.

## 11. 개발 로드맵

1. **뼈대**: Next.js + Tailwind + Supabase 연결, 로그인(공유 비번) 미들웨어
2. **데이터 모델**: 테이블/버킷 생성 + seed(위 스케줄)
3. **갈 데이트 CRUD**: 코스/장소 추가·수정·순서변경, 타임라인 뷰
4. **이동시간 엔진**: 지오코딩 캐시 + 차/대중교통 API + 비는시간 계산 & 경고
5. **간 데이트**: 상태 전환 + 사진 업로드 갤러리
6. **홈 피드 & 감성 UI 다듬기**
7. **배포**: Vercel + 환경변수

## 확정된 결정 (v3)

- 이동수단 **고정 안 함** — 구간마다 차/대중교통 둘 다 표시, 조정은 사람이.
- **머무는 시간(stay_min)** — 선택 입력. 넣으면 여유 계산에 반영.
- **협업 앱** — 둘 다 일정 조정·사진 업로드·리뷰 가능.
- **멤버(이름)** — 공유 비번 통과 후 이름 정함, 변경 가능. 사진·리뷰의 작성자.
- **서로 리뷰** — 한 데이트에 사람당 별점+한줄평 1개씩(내 리뷰/상대 리뷰 나란히).
